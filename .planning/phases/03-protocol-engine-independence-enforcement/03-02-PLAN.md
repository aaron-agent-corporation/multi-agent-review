<objective>
Make the RED `mar run` anchor GREEN: build the XState v5 protocol engine that drives an input document through all 6 phases with enforced turn-taking, an artifacts-on-disk gate between phases (PROT-03), structural draft independence via the Plan-01 scope primitives (PROT-04), and a thin `mar run <input>` CLI subcommand (PROT-01). This is the core vertical slice — after it, a user can run the full protocol end-to-end on a document.

Purpose: PROT-01/03/04 land together as one observable capability ("watch a doc progress through 6 gated phases"). The engine reuses every existing primitive (turn seam, manifest, naming, gate-style) unchanged; only the XState control-flow shell is net-new (user-ratified over the RESEARCH sequential-loop recommendation — see 03-PATTERNS.md ratification note).

Concurrency decision (explicit): the per-phase fan-out uses **bare `Promise.allSettled`, NO `p-limit` this phase**. RESEARCH recommends `p-limit` for larger rosters, but for the ≥2-vendor MVP roster bare `allSettled` is sufficient (no need to throttle 2–3 processes) and avoids an unvetted install. `xstate@5` is therefore the ONLY new package this phase (Task 0 legitimacy gate); `p-limit` is deliberately NOT installed. If a future phase grows the roster, revisit adding `p-limit@^7` behind its own legitimacy gate.

Engine-is-source-of-truth decision (explicit): the seq each agent uses is decided ONCE, in the engine's fan-out (via `nextSeq`), and the fan-out returns the EXACT artifact paths it wrote. The phase gate is checked against THAT path array — the gate never independently recomputes seqs or guesses paths. This prevents the gate from checking an unwritten path (false fail/pass). `expectedPhaseArtifacts` is a helper for tests/derivation only; the engine's live gate input is always the collected written paths.

Output: `xstate@5` installed (only new package); `src/protocol/{phases,gate,engine}.ts`; `mar run` wired in cli.ts; gate + engine unit tests; the Plan-01 e2e anchor green.
