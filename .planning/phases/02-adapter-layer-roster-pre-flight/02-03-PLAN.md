RESEARCH Pattern 4 (MarConfig schema — implement this shape):
```typescript
const Base = { name: z.string().min(1), bin: z.string().optional(), model: z.string().optional(),
  timeoutMs: z.number().int().positive().optional(), extraArgs: z.array(z.string()).optional() };
const Agent = z.discriminatedUnion("vendor", [
  z.object({ vendor: z.literal("claude"), ...Base }),
  z.object({ vendor: z.literal("codex"),  ...Base }),
  z.object({ vendor: z.literal("gemini"), ...Base }) ]);
const MarConfig = z.object({ agents: z.array(Agent).min(1),
  defaults: z.object({ timeoutMs: ...default(600000), retries: ...min(0).default(2) }).default({}) })
  .superRefine(/* duplicate-name check */);
```

RESEARCH Pattern 5 (onPath — PATH walk, no shell):
```typescript
function onPath(bin): string | undefined  // walk process.env.PATH.split(delimiter), existsSync, PATHEXT on win32
```

## Tasks — CONTEXT.md decision citations

- **D-18:** Roster lives in `mar.config.json` at project root; JSON, zod-validated (no YAML). (Task 1)
- **D-19:** Agent entries are structured fields `{name,vendor,bin?,model?,timeoutMs?,extraArgs?}` + `defaults` block; vendor selects adapter; bin overrides binary; extraArgs append-only — adapters own command shape. (Task 1)
- **D-20:** `mar invoke --agent <X>` resolves against roster entry NAMES only; missing file -> clear error; single resolution path (`resolveAgent`). (Task 1)
- **D-21:** Ship `mar init` — probes PATH for claude/codex/gemini and writes a starter `mar.config.json` with detected vendors. (Task 3)
- **D-29:** <2-distinct-vendor gate: HARD refusal at run start, no override; error names vendors found; `mar invoke` exempt. (Task 2)
- **D-30:** Partial-failure `--skip-failed` drops failing agents and proceeds ONLY if >=2 distinct vendors remain healthy. (Task 2)
- **D-34:** `mar.config.json` is committed to the repo (not gitignored) — roster is project configuration. (Task 3 — starter config is a tracked project file; do NOT add it to .gitignore)

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| mar.config.json (on-disk, committed) -> roster fields -> adapter argv | config supplies bin/extraArgs that become argv elements |
| process.env.PATH -> init detection | PATH entries determine which binaries are "detected" |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-09 | Tampering / EoP | malicious bin/extraArgs in config | accept (documented) | `bin` running a binary is the intended test-injection feature; `mar.config.json` is TRUSTED project config (committed, D-34). zod enforces extraArgs is `string[]`; values are append-only argv elements, never eval'd/interpolated into a shell |
| T-02-10 | Tampering | config validation | mitigate | MarConfig.parse rejects unknown vendor / dup name before any value reaches an adapter |
| T-02-11 | EoP | init PATH detection | mitigate | PATH-walk via existsSync only — no shell, no `which` spawn (no-shell-injection posture) |
| T-02-12 | Spoofing | single-vendor review bypass | mitigate | assertReviewable hard gate, no override flag; single-vendor review is out of scope (PROJECT.md) |
| T-02-SC | Tampering | npm installs | mitigate | none added this phase (D-35) |
</threat_model>
